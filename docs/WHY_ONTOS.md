# Why .ontos

Markdown is excellent for readable pages. Many project documents, however, now
need to act as structured context for humans and AI tools at the same time.

`.ontos Protocol` focuses on project context that needs:

- stable node IDs
- explicit fields
- references between decisions and verification steps
- clear AI boundaries
- deterministic formatting
- Git-friendly review
- local-first usage

## Why Not Just Markdown?

Markdown is optimized for prose. Project state often needs a durable structure:

- a task should keep the same ID after edits
- a review pack should include boundaries and risks
- a release plan should have checkable verification steps
- an AI handoff should include only the relevant node path and fields

`.ontos` keeps plain text while making those structures explicit.

## Why Not A Database?

Project context should be portable and inspectable. A `.ontos` file can live in
Git, travel with a repository, and remain readable without a service account.

## Why Now?

AI coding tools work better when context is bounded, structured, and
verifiable. `.ontos` gives teams a shared text protocol for that context without
locking them into a hosted app.
